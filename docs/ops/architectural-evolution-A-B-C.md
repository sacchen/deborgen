This document outlines the three major architectural upgrades applied to `deborgen` to transition it from a simple "v0" loop into a functional, safe, heterogeneous compute cooperative. 

These changes were designed to support real-world usage among friends donating their personal hardware (laptops, gaming PCs, cloud VMs) without requiring complex production orchestrators like Kubernetes.

---

## Phase A: Artifact Storage via S3 Presigned URLs

### The Problem: Coordinator Bottlenecks
In our original hub-and-spoke architecture, the central coordinator was a bottleneck. If 10 workers finished deep learning jobs simultaneously and tried to `POST` 1GB of result files to the coordinator, the lightweight coordinator (e.g., a $5/month Droplet) would run out of memory, disk space, and network throughput. It would likely drop HTTP connections, causing routine worker heartbeats to fail and jobs to incorrectly be marked as "lost."

### The Solution: Offloading Bandwidth via Presigned URLs
To keep the coordinator small, we integrated **S3 Presigned URLs**.
1. **The Ticket Agent:** The coordinator acts merely as a secure ticket agent. It holds the secret S3 credentials (via `boto3`).
2. **The Request:** When a worker finishes a job, it asks the coordinator: *"I have an artifact named `artifacts.zip`, where can I put it?"*
3. **The URL:** The coordinator generates a temporary, cryptographically signed URL and returns it to the worker.
4. **Direct Upload:** The worker uses standard HTTP to `PUT` the large file *directly* to the massive cloud provider (AWS S3, Cloudflare R2), completely bypassing the coordinator's network.

### Educational Takeaway
**Data Plane vs. Control Plane:** By separating the Control Plane (the coordinator tracking state) from the Data Plane (the actual heavy files moving to S3), we can scale the cluster horizontally while keeping the central orchestrator practically free to run.

Additionally, we updated the worker to run jobs inside isolated `tempfile.TemporaryDirectory` paths. This ensures the user's host machine stays clean and guarantees that any artifacts collected were strictly generated by that specific job run.

## Phase B: Heterogeneous Label-Based Routing

### The Problem: Blind FIFO Queues
Initially, the coordinator simply took the oldest job in the `queued` state and handed it to the first worker that asked. This works fine if all nodes are identical. However, in a cooperative cluster, one friend might have an `x86_64` machine with an RTX 3060 GPU, while another might have an `arm64` MacBook M1. If a job requiring a GPU is handed to the MacBook, it will fail.

### The Solution: Subset Matching & Auto-Detection
We transformed the scheduler to be "label-aware":
1. **Requirements:** Jobs can now declare a `requirements` dictionary (e.g., `{"gpu": "rtx3060", "os": "linux"}`).
2. **Auto-Detection:** To reduce friction, the worker agent (`agent.py`) now uses Python's `platform` and `os` modules to automatically detect the host's OS, CPU architecture, and core count upon startup.
3. **Subset Matching:** When a worker polls for a job, the coordinator fetches the queue and evaluates it against the worker's labels in Python. A job is only assigned if the worker's labels are a complete *superset* of the job's requirements.

### Educational Takeaway
**Python vs. SQL for Complex Logic:** We chose to fetch the oldest queued jobs and perform the JSON subset matching in Python, rather than attempting to write a complex, multi-layered SQL query using SQLite's `JSON1` extension. While SQL *can* do it, implementing the match in Python keeps the codebase readable, easily testable, and robust across varying SQLite versions on different host OSs. We maintain database atomicity by performing a targeted `UPDATE ... WHERE id = ?` once the match is found.

## Phase C: Safe Participation (Time-of-Day Schedules)

### The Problem: Disruption of Personal Machines
If friends are donating their personal gaming PCs to the cluster, the worker agent must not ruin their primary computing experience. If a heavy simulation starts while they are playing a game or working, they will simply uninstall the agent.

### The Solution: `--work-hours`
We introduced a command-line flag to the worker agent: `--work-hours "22:00-08:00"`.
1. **Time Boundaries:** The worker loop evaluates the current local time against this window.
2. **Resting State:** If the current time is outside the permitted window, the worker skips polling the queue for new jobs.
3. **Persistent Heartbeats:** Crucially, even while "resting," the worker continues to send regular HTTP heartbeats to the coordinator.

### Educational Takeaway
**Cluster Visibility:** Why keep sending heartbeats if the node isn't accepting work? If a worker simply disconnects during its quiet hours, the cluster loses visibility into total capacity. By continuing to heartbeat, the coordinator knows the machine is alive, properly configured, and will return to the active pool at its scheduled time. This architectural choice paves the way for future advanced scheduling (e.g., "Job X requires a GPU, so it will remain queued until 22:00 when 3 GPUs wake up").
