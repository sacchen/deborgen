# Gemini Context & Project Memory

This file serves as a memory core and context guide for AI assistants working on `deborgen`.

## Project Overview
`deborgen` is a small, cooperative, distributed compute system for friends. It allows users to pool their idle personal machines (gaming PCs, laptops, droplets) to run embarrassingly parallel workloads (simulations, optimization sweeps). It explicitly avoids production orchestrator complexity (like Kubernetes or Borg) in favor of a lightweight, understandable, and trusted model.

## Core Architecture (v1)
- **Coordinator (`src/deborgen/coordinator/app.py`):** A lightweight FastAPI control plane backed by SQLite. It maintains the authoritative state of the job queue, node labels, leases, and artifacts.
- **Worker Agent (`src/deborgen/worker/agent.py`):** A pull-based Python client that runs on participant machines. It auto-detects hardware, polls for matching jobs, executes them in isolated subprocesses, and uploads results.

## Key Architectural Decisions & Features
1. **Pull-Based Matchmaking:** Workers poll `/jobs/next`. The coordinator evaluates the queue against the worker's auto-detected labels (`os`, `arch`, `cpu_cores`) and user-provided labels (e.g., `gpu: rtx3060`). It assigns jobs only if the worker's labels are a complete superset of the job's `requirements`.
2. **S3 Artifact Offloading:** To prevent the central coordinator from becoming a bandwidth bottleneck, workers upload job artifacts directly to cloud storage (AWS S3, Cloudflare R2) using Presigned URLs generated by the coordinator via `boto3`.
3. **Safe Participation (Work Hours):** Participants can pass `--work-hours "22:00-08:00"` to the agent. The agent will pause polling for jobs outside these hours but will continue to send persistent heartbeats so the cluster retains accurate capacity visibility.
4. **Isolated Execution:** Jobs run inside `tempfile.TemporaryDirectory` paths. When the job finishes, the directory is automatically zipped and uploaded, ensuring deterministic artifact collection and leaving the host machine clean.

## Future Strategic Direction
- **The Go Hybrid Approach:** There is an educational plan (`docs/ops/go-hybrid-plan.md`) to rewrite the Worker Agent and CLI in Go to allow zero-dependency single-binary distribution to friends, while keeping the Coordinator in Python for rapid API/scheduling iteration.
- **Operations Research:** Future phases aim to use Python's data ecosystem to build optimization-driven scheduling and fair-share allocation algorithms based on the history stored in SQLite.

## Operational Norms
- Prefer SQLite/Python simplicity for the coordinator.
- Zero or low friction for worker setup is paramount.
- Maintain tests via `uv run pytest`.
- Keep changes human-scale and explainable.
